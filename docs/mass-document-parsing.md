# Архитектура массовой обработки и валидации документов (RAG)

Этот документ описывает планируемую архитектуру для автоматизированного сканирования, парсинга и валидации большого объема технической литературы (10 000+ документов) для системы RAG (Retrieval-Augmented Generation).

## 1. Автоматизированный конвейер парсинга (Data Ingestion Pipeline)
Обработка десятков тысяч PDF-документов требует выделенной асинхронной инфраструктуры, чтобы не перегружать основной Node.js/Next.js портал и не разориться на внешних платных API.

*   **Очередь задач (Message Queue):** Использование RabbitMQ или Redis (BullMQ). Пользователь (или скрипт) массово загружает ссылки на исходные PDF-файлы в очередь.
*   **Python Worker (Парсер):** Выделенный микросервис на Python, который берет задачи из очереди. Под капотом использует **Docling** (от IBM) или **MinerU** для полностью локального и бесплатного распознавания.
    *   *Задачи воркера:* Определение структуры страницы, OCR встроенных изображений, извлечение таблиц и слияние всего контента в чистый формат Markdown.
*   **Хранилище данных (Data Lake):** Использование S3-совместимого хранилища (например, MinIO).
    *   Сырые PDF-файлы сохраняются в один бакет.
    *   Сгенерированные Markdown-файлы и извлеченные картинки (чертежи) сохраняются рядом.

## 2. Модуль валидации документов (Human-in-the-Loop)
ИИ делает 95% работы, но перед добавлением документа в векторную базу данных для поиска, операторы должны иметь возможность проверить и поправить огрехи распознавания (например, сложные формулы или специфические сокращения).

*   **Интерфейс валидатора (Next.js):** Отдельный закрытый раздел в основном портале (например, `/admin/documents-validation`).
*   **Двухколоночный UI:**
    *   *Левая колонка:* Просмотрщик оригинального PDF (с поддержкой зума и постраничной навигации).
    *   *Правая колонка:* Интерактивный Markdown-редактор с загруженным текстом от ИИ.
*   **Жизненный цикл документа:**
    1.  Python-воркер завершает парсинг и сохраняет Markdown в базу со статусом `Ожидает проверки`.
    2.  Оператор открывает документ, сверяет текст с левой колонкой, вносит правки в редакторе.
    3.  Оператор нажимает **«Утвердить»**. Документ получает статус `Проверено` и отправляется на следующий этап (Чанкинг и Эмбеддинги).

## 3. Интеграция с RAG
После утверждения оператором, `Проверенные` документы:
1.  Разбиваются на смысловые куски (чанки) с помощью интеллектуальных сплиттеров (например, `MarkdownHeaderTextSplitter`), чтобы не разрывать контекст заголовков.
2.  Превращаются в векторы с помощью локальной модели эмбеддингов.
3.  Загружаются в промышленную векторную базу данных (например, **Milvus** или **Qdrant**), где становятся доступны для поиска через чат-ботов и умных ассистентов.
